# ADR-0006 — Role-based access control (RBAC)

- **Status:** Accepted (direction). Role catalogue & scoping granularity are iterative.
- **Date:** 2026-07-24
- **Depends on:** [ADR-0004](0004-identity-and-credential-model.md) (host is the final
  arbiter; RBAC is a second gate), [ADR-0005](0005-typed-operation-protocol.md)
  (operations carry `required_permission`), [ADR-0002](0002-postgresql-control-plane.md)
  (storage, `organization_id`).

## Context

v0.x has one coarse split: **admin** (in `sudo`/`wheel`/…) vs **read-only**. For "many
users, multiple administrators, B2B" (TENODERA_V2) that is far too blunt — you cannot
express "Linux operator who may restart services but not manage users", "security
auditor who may only read audit + inventory", or "operator scoped to the `payments`
host group".

RBAC must not be confused with the host's decision. ADR-0004 already settled that:
**the host's `sudo`/HBAC is the final arbiter**; RBAC is a *panel-side* policy gate.

## Decision

### Two gates, RBAC can only restrict
A privileged operation must pass **both**:
1. **Tenodera RBAC** — is this operator allowed to *attempt* this operation on this
   host through the panel? (least-privilege within the organization)
2. **The host** — `sudo` Sudo Rules / HBAC actually permit it as that user? (ADR-0004)

RBAC **cannot grant beyond the host** — a user with full sudo on a host but only
`service.view` in Tenodera still cannot restart it via the panel. RBAC only ever
*narrows*. This makes a compromised control plane unable to exceed host policy.

### Permissions are the operation namespace
**Permissions map to the operation taxonomy of ADR-0005**, not a parallel list that
can drift. Each operation declares its `required_permission`; permissions exist at two
granularities:
- **fine** — one per operation (`service.restart`, `package.install`, `file.write`);
- **coarse** — a grouping (`service.manage` ⊇ start/stop/restart/enable/disable),
  for ergonomics.
A role holds a set of permissions (fine and/or coarse).

### Roles, bindings, scoping
```
roles                      (id, org_id, name, builtin)
permissions                (key)                     -- = operation keys + groups
role_permissions           (role_id, permission)
user_roles                 (user_id, role_id, scope) -- scope below
```
- **Scope** attaches a role grant to a target: **global**, a **host**, a **host group**,
  or a **tag selector** (`environment=production`). Tag scoping is the scalable form —
  "Operator on everything tagged `team=payments`".
- Ship **built-in roles** (from TENODERA_V2): `Administrator` (all), `Operator`
  (view + service control + journal), `Security Auditor` (view + audit + inventory),
  plus fully **custom roles**.
- **Deny-by-default:** no matching allow ⇒ denied. First release is **allow-only**
  (no explicit deny rules) to keep evaluation simple and predictable.
- Every table carries `organization_id` (ADR-0002); one org today, RLS later.

### Enforcement point
The **operation router** (ADR-0005) checks `required_permission` against the actor's
effective permissions **for the target host's scope** *before* dispatch — the same
structural "can't forget the check" property as the operation-policy enforcement.
The UI also hides actions the operator lacks, but the router is the boundary; the UI
is convenience.

### High-risk step-up is orthogonal
The high-risk step-up + fresh MFA (ADR-0004) is a **separate control** layered on top:
having the permission is necessary but not sufficient for a high-risk op — step-up is
still required. RBAC says *who may*, step-up says *prove it now*.

## Rejected alternatives

- **Keep coarse admin/readonly.** Cannot express per-subsystem or per-host-group
  authority for a multi-admin fleet. Rejected (retained only as two default roles).
- **RBAC as the sole authority** (ignore host). A compromised server could act as any
  operator. Rejected — host stays final (ADR-0004).
- **A parallel permission taxonomy** decoupled from operations. Guarantees drift and
  missed gates. Rejected — permissions ARE operation keys.
- **A general policy engine (OPA/Rego, ABAC) now.** Powerful but heavy for v1; RBAC +
  tag scoping covers the stated needs. Deferred; the schema doesn't preclude it later.

## Consequences

- Adding an operation (ADR-0005) automatically implies a permission — the taxonomies
  can't diverge. New op ⇒ decide its permission (fine key + which coarse group) as
  part of adding it.
- Tag-based scoping needs the inventory/host-tag model (TENODERA_V2) and makes host
  onboarding drive authorization ("new prod host inherits Operator grants for
  `environment=production`") — powerful, but tag integrity becomes security-relevant.
- Role/permission changes are **audited** (ADR-0007) — `permission_changed` is a
  security event.
- More model than v0.x, but it's the minimum for B2B multi-admin; UI needs a role
  editor and a per-user effective-permissions view.

## Open questions

1. **Scoping granularity for v1:** global + host + host-group is table-stakes; do we
   ship **tag-selector** scoping in v1 or defer to after the inventory model lands?
2. **Coarse permission groupings** — the exact catalogue (which ops group together).
3. **Role delegation / admin-of-admins** — can a non-global role manage other users'
   roles within its scope, or is user/role administration global-only in v1?
4. **Effective-permission evaluation** across overlapping scoped grants (union;
   confirm no deny-precedence needed while allow-only).

## Acceptance criteria

- An operator with `service.view` but not `service.manage` is refused `service.restart`
  by the **router** (not just hidden in the UI), even on a host where their `sudo`
  would allow it — proving RBAC narrows.
- A role scoped to host-group A cannot act on a host in group B.
- Every operation's `required_permission` resolves to a real permission; a new
  operation with no permission mapping fails a startup/CI check (no ungated ops).
- Granting/revoking a role writes a `permission_changed` audit event.
- Built-in roles exist out of the box; a custom role can be created, scoped, and
  enforced end-to-end.
